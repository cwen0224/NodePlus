# 本地 Web UI 協同開發準則（v1.1）

## 1. 目的
- 建立一套可持續、可回退、可驗收的本地 Web UI 開發流程。
- 降低溝通成本，避免需求反覆與隱性耦合。
- 確保每次修改都可被驗證，並能快速定位問題。

## 2. 專案範圍與非目標
- 範圍：本地執行的 Web UI、前後端串接、基本資料流程、錯誤處理與可觀測性。
- 非目標：未定義完成條件的功能、一次性大改全部模組、無驗收標準的 UI 微調。

## 3. 協作角色與責任
- 我（Codex）負責：需求拆解、技術方案、實作修改、風險提示、測試與回報。
- 你（Sang）負責：需求確認、產品取捨、優先級決策、最終驗收。
- 不明確需求處理：我會先列出假設並標記風險，先做最小可行版本（MVP）。

## 4. 需求輸入規範（你給我的內容）
每次需求至少包含以下資訊：
- 使用者角色（誰在用）
- 目標（想完成什麼）
- 操作流程（按鈕、頁面、狀態）
- 驗收標準（如何判定成功）
- 優先級（高 / 中 / 低）
- 時間限制（若有）

未提供完整資訊時，先補齊「目標 + 驗收標準」再進入開發。

## 5. 任務拆分與開發節奏
- 以 3 到 7 天為一個迭代。
- 每個迭代只做 1 到 3 個可驗收功能。
- 每個功能限制在單一明確目的，避免混合重構與新功能。
- 每輪結束固定產出：Demo、問題清單、下一輪優先級。

## 6. 架構與解耦原則（強制）
- 每個功能需可獨立開發、獨立測試、獨立替換。
- 模組間僅透過明確介面溝通（API / 事件 / 型別）。
- 禁止跨模組直接讀寫內部狀態。
- 共用邏輯要抽成共用模組，不允許複製貼上長期維護。
- 若改 A 功能需連動改多個無關模組，先重構再擴充。

## 7. UI/UX 原則
- 先可用再精緻：先完成流程，再做視覺優化。
- 桌面與行動裝置都要可操作（RWD）。
- 關鍵操作需有狀態回饋（載入中 / 成功 / 失敗）。
- 破壞性操作（刪除、覆蓋）必須二次確認。
- 表單錯誤要即時且可理解，避免只顯示通用錯誤。

## 8. 程式碼規範
- 命名清楚，避免語意不明縮寫。
- 單次修改聚焦單一目的（可追蹤、可回滾）。
- 複雜邏輯加「為什麼這樣做」的簡短註解。
- 不提交未使用程式碼、註解垃圾、臨時 debug 殘留。

## 9. Git 與變更管理
- 分支命名：`feat/*`、`fix/*`、`refactor/*`、`chore/*`。
- Commit 訊息：`type(scope): summary`（例如 `feat(editor): add save hotkey`）。
- 需求變更需記錄：改了什麼、為什麼改、影響範圍。
- 大改前先確保可回退（備份或可還原版本）。

## 10. 測試與驗收門檻（Definition of Done）
功能完成需同時滿足：
- 正常流程可用。
- 錯誤流程可預期處理。
- 至少一個邊界情境已驗證。
- 不影響既有功能（回歸確認）。
- 相關測試通過（至少單元測試 + 基本流程測試）。

若測試失敗，該功能不得標記完成。

## 11. 偵錯模式與日誌規範（強制）
- 系統需提供可切換 Debug Mode。
- 開啟 Debug 時至少記錄：操作步驟、API 請求/回應摘要、錯誤堆疊、耗時。
- 正式模式僅保留必要日誌，避免效能負擔。
- 開發環境預設開啟 Debug，正式環境預設關閉。
- 日誌不得暴露密碼、金鑰、個資，必要時遮罩。

## 12. 安全與設定管理
- 禁止將密碼、Token、金鑰硬編碼在前端或 repo。
- 所有外部輸入都需驗證與過濾。
- 重要資料操作需權限控管。
- 啟動前確認環境變數完整且來源正確。

## 13. 本地執行與重啟規範
- 每次修改完成後，需重啟本地服務並確認可正常啟動。
- 回報需包含：
  - 是否成功啟動
  - 若失敗：錯誤摘要與可能原因
  - 下一步處置

## 14. 每次修改後固定回報格式（我對你）
- 已修改內容摘要
- 自動測試結果（通過 / 失敗）
- 手動冒煙測試清單
- 風險與後續建議（若有）

若尚未建立自動測試，需明確標註測試缺口與最低補強方案。
